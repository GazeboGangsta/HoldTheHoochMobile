# Session retrospective — 2026-04-20

Honest look at what worked and what didn't this session. Paired with [docs/HANDOVER-2026-04-20.md](../../HANDOVER-2026-04-20.md) (session scope + state) and the standing [CLAUDE.md Workflow section](../../../CLAUDE.md#workflow) (the rules this retro graded against).

## Session in one line

Closed M4 properly, polished M5 heavily, landed CI/CD, shipped the leaderboard scene — at the cost of rebuilding controls twice because mobile gesture realities weren't sanity-checked up-front.

## What went well

### The formalized workflow actually paid out

- `systematic-debugging` + `test-driven-development` caught the `ScorePopup` crash root cause from the *failing test output* rather than guesswork. The reviewer's earlier guess about `OpacityEffect` was wrong; the actual crash was `MoveByEffect` — we only discovered that because we wrote a test that reproduced the crash first.
- `writing-plans` before M4b and before the leaderboard meant the work took two well-structured passes with minimal rework. The plans were detailed enough that subagents could execute tasks without asking clarifying questions.
- `requesting-code-review` at end of feature (leaderboard) caught two real Important issues the per-task reviews missed (empty-state refresh gap, ignored `error` parameter).

### Subagent-driven development scaled

Per-task implementer + spec reviewer + quality reviewer = consistent output quality. A fresh subagent per task meant no context pollution, and the two-stage review surfaced things the implementer's self-review missed. The one fix-subagent round we needed (empty-state pull-to-refresh) was resolved cleanly.

### STATUS.md as living ground truth

Whenever code and docs drifted — and they did, twice, quietly — the drift got caught within the same session via review pass and closed in a follow-up commit. Future-session agents won't inherit stale claims. Worth the habit.

### CodeMagic setup went faster than expected

First API-triggered Android build failed (instance type), user enabled billing, one commit later both Android and iOS passed on the very next attempt. Having `codemagic.yaml` as source-of-truth meant the config was version-controlled from day one.

## What didn't go well

### The control scheme was rebuilt three times

Session started with "drag left half to balance" (the original web-game-inspired design). Hit:

1. System back-gesture conflicts on Android/iOS screen edges.
2. Multi-touch bug: lifting the balance finger cancelled jumps.
3. Sign inversion: drag-right made tankard go left (counter-intuitive for "follow finger" mental model).

Then switched to floating tilt buttons in bottom-left corner. That surfaced:

4. Wanting a dedicated jump button too, not a tap-right-half fallback.
5. The play area didn't need to be full-screen; a control strip is cleaner.

**Lesson:** should have sketched the mobile control layout as a brief design doc before coding *any* input. The web-game controls (keyboard arrows) didn't translate 1:1 and we discovered that the expensive way — ~5 separate fix commits for what should have been one planning pass.

### Drift bias was a late discovery

Spec (`GAME_DESIGN.md`) calls for "natural wobble + bias toward outer edges as speed increases." Only implemented the wobble half through M4b Task 3 — tests passed because they only checked that amplitude scaled with difficulty. Player feedback on device revealed the spill meter never filled: peak tilt from pure wobble is geometrically ~0.07, threshold is 0.7.

**Lesson:** test specs against *gameplay outcomes*, not just unit-level math. A test like "within 30s at difficulty 1.0, does tilt cross spillThreshold without player input?" would have caught the missing drift at Task 3 time, not three commits later.

### Doc drift snuck in twice

Same pattern both times: a focused commit closed the specific items it was about, but adjacent sections (Known Issues table, milestone summary, "What's playable today") were left stale because they weren't in the commit's explicit scope. Code-quality review caught both — but the cleaner fix would be a docs-drift checklist on any commit that touches milestone status.

**Lesson:** when marking a milestone complete, scan for every mention of the milestone in STATUS + ROADMAP + design docs in the same commit, not just the specific checkbox.

### Reviewer cried wolf once

The leaderboard final review flagged "ApiClient lifecycle / http.Client leak" as an Important issue. Verified empirically: `ApiClient` uses top-level `http.get`/`post` which delegate to a package-managed shared `Client` — no per-instance resource, no leak. Five minutes lost verifying, but the cost of *not* verifying would have been bigger.

**Lesson:** reviewers aren't oracles. Each Important issue deserves a quick verify-from-code step before fixing.

### API token in chat transcript

Shared a CodeMagic API token directly in the chat to bootstrap CodeMagic integration. Convenient at the time; now needs rotation. Flagged in `SIGNING.md` and in the handover. Would be better handled next time via a local `.env` read rather than paste-in-chat.

## Tooling observations

- **`ScheduleWakeup`** was heavily used for "wait on build" pacing. Worked well. One anti-pattern: scheduling a wakeup *and* the user responds before it fires creates an awkward stale wakeup. Fine to absorb, just note it.
- **Wireless debugging over adb** worked reliably across two phones. Pair-and-connect is a two-step flow that's easy to forget; documented in the handover.
- **On-screen diagnostic HUD** (live `TextComponent` showing `gurglesY`, `onGround`, `tilt`, etc.) was dramatically more useful than chasing `debugPrint` through `adb logcat`. Reinforced the gotcha already logged.
- **Background `flutter build apk` + `ScheduleWakeup`** pattern means I can keep reviewing/writing while builds run. Worth formalizing as a convention — anytime a build is expected to run >30s, fire-and-forget it.

## What to do differently next session

1. **Before any input/control change, sketch the intended mobile layout** as 3–4 lines of prose: "which button does what, where does it sit, what's the hit-testing priority." Catches gesture conflicts and multi-touch scenarios before they're coded.
2. **Gameplay-level tests** alongside unit tests: "does the game reach this gameplay state with no player input in N seconds?" The ratio of unit-test coverage to *felt-gameplay* coverage was off this session.
3. **Commit messages explicitly list STATUS/ROADMAP/design docs touched.** Makes drift visible in log review.
4. **API tokens go through local `.env` not chat paste.** Already implicit; worth making explicit as a working agreement.
5. **Retros as a standing practice** at the end of each session where a milestone closes, not just at epic boundaries. Catches process wobbles while they're fresh.
