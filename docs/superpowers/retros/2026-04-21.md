Long session spanning 2026-04-20 into 2026-04-21. Paired with [docs/HANDOVER-2026-04-21.md](../../HANDOVER-2026-04-21.md) (session scope + state) and the standing [CLAUDE.md Workflow section](../../../CLAUDE.md#workflow) (the rules this retro grades against).

## Session in one line

Closed M4c and M5a via subagent-driven development, shipped the first IPA to TestFlight — at the cost of ~2 hours fighting CodeMagic's broken "automatic" iOS signing before falling back to a manual-upload flow that actually works.

## What went well

### Subagent-driven development scaled across two full milestones

M4c (six tasks + three post-review fixes) and M5a (nine tasks + three post-review / tune fixes) both executed cleanly through the implementer → spec reviewer → code quality reviewer loop. A few moments where the process really earned its keep:

- **M5a Task 4 post-review caught a real downstream trap.** The Task 4 implementer made `ParticleSystemComponent`s children of `SplashEmitter` (sensible architectural choice for the test-harness concurrent-mod issue) but didn't update `reset()` to clear them. Both reviewers independently flagged that Task 6's cleanup loop on `GameScene.children.whereType<PSC>()` would silently miss them. A fix commit (`d097418`) added the cleanup, and Task 6's test was written with `descendants()` instead of `children`. Without those reviewer passes, the restart integration test would have given a false-green while splash particles leaked across runs.

- **M5a Task 7 post-review caught `isMounted` vs `isAttached`.** Trivial-looking one-line change (`if (!isMounted)` → `if (!isAttached)`) but with real semantic consequence — `FlameGame.isMounted` stays true after `Navigator.pop` disposes the `GameWidget`, so the `Future.delayed` callback could have fired against a dead game. Only the code reviewer noticed; implementer self-review didn't catch it.

- **M4c Task 5 post-review caught "the floor is too loose."** Implementer believed the 90 px full-hold-jump floor had ~2.6 px headroom. Code reviewer actually instrumented the simulation and measured ~208 px (~118 px headroom), which meant the test would silently tolerate a regression from 208 → 95. Floor raised to 150 px. The implementer's mental model was just wrong about Flame's velocity-ramp semantics; only running the numbers caught it.

### Design review before plan review before implementation paid off

For M5a, the `brainstorming → writing-plans → subagent-driven-development` chain moved fast because each stage caught its own class of issue:

- Brainstorming locked the three major design axes (splash hybrid pattern, intensity + game-over burst, sparkle per-kind scaling) before any code was imagined.
- Writing-plans surfaced two bugs in the spec itself — frame-rate-dependent decay math and an `int` constant typed as `double`. Fixed inline before dispatching implementers.
- Spec-reviewer + code-reviewer then caught what I've already written above.

The signal: each review stage catches a different class of problem, and running them in this specific order keeps the cost-per-catch low.

### Device playtest as a named plan task

M5a Task 8 was explicitly "install on device + play through the four feel moments + tune." That caught the splash-position bug (particles emitting 40–140 px away from the rim because of a `bottomCenter`-anchor math mistake) that no unit test would have caught — the stub `_StubGurgles` returned a fixed position that hid the real transform error. Worth keeping "device playtest" as a named task (with documented acceptance criteria) in future gameplay milestones, not an afterthought.

### CodeMagic REST API for monitoring

Once it became clear we'd be doing many CodeMagic iterations, polling builds via `curl` + `python -c 'print(json.load(...))'` + `until ... do sleep 45; done` was dramatically faster than context-switching to the browser dashboard. Also gave us programmatic access to build logs (which is how we finally saw the `Cannot save Signing Certificates without certificate private key` error that the web UI had buried). Worth formalising — a small helper script in `tools/codemagic-monitor.sh` or similar for next time.

## What didn't go well

### CodeMagic's "automatic iOS code signing" is broken for first-time accounts

~2 hours of wall-clock time burned on this. The failure modes are subtle and the error messages are misleading:

- `ios_signing` declarative block runs a strict pre-flight profile-existence check that rejects the build before any script can run. Docs imply it auto-creates; in reality it only *looks up*.
- `app-store-connect fetch-signing-files --create` silently fails to save the generated private key (`"Cannot save Signing Certificates without certificate private key"`), leaving an orphaned cert on Apple's side that can't be used for signing on subsequent runs.
- Revoking the orphaned cert and retrying reproduces the same private-key-save failure. It's a genuine CodeMagic CLI bug, not a configuration error.

**Lesson for future CI-tool onboarding**: when a documented automatic-setup feature fails on first run, STOP trying variations and fall back to the fully-manual path within 20 minutes. We tried five different yaml configurations (declarative `ios_signing`, script-based `fetch-signing-files --create`, script-based with both `IOS_APP_DEVELOPMENT` + `IOS_APP_STORE`, `certificates create --save`, env-var-based private-key injection) before accepting that the automatic flow just doesn't work and doing the OpenSSL + Apple portal + CodeMagic-team-store manual upload. That working path took 15 minutes once started. If we'd started there we'd have shipped an hour earlier.

### I pushed speculative yaml fixes repeatedly

Each yaml iteration above was committed, pushed, and retagged before the next was tried. That's 5 commits of churn that mostly don't reflect real state. Should have either:

- Kept local-only iterations until one worked (nicer git history, but no feedback loop against CodeMagic without pushing).
- Batched all attempts into one commit with explicit documentation ("tried A, failed; tried B, failed; settled on C").

As-is, the session's git log has a noisy middle section. The retro-lesson "make commits that future-you can read" wasn't honored.

### User frustration was a valid signal I almost missed

User's "genuinely what is the point of CodeMagic if it can't handle this?" was the moment to stop and reassess strategy, not push another yaml fix. I was one edit away from committing yet another speculative change when the user spoke up. Their frustration was correct signal that the overall approach was wrong — I was in the tunnel of "one more fix might work" instead of zooming out to "what's the fastest thing that DEFINITELY works?"

**Lesson**: when a user signals frustration at cumulative time sunk on a single issue, that's the prompt for a strategy reset, not a process continuation. The honest response was "you're right, we've been chasing ghosts; let me revert everything and pick the dumbest-simplest working path" — which is what I then said, and which worked immediately.

### The monitoring loop once blocked on a `sleep 30` chain

My `sleep 30 && curl ...` attempt was blocked by the harness's anti-sleep rule. Switched to `until ... do sleep 45; done` which is the sanctioned polling pattern. Minor speedbump. But reminds: the harness actively enforces "use the right pattern for waiting" — don't try to work around it with chained sleeps.

## Tooling observations

- **`git tag -f v0.1.0 HEAD && git push -f origin v0.1.0`** was used five times this session to keep the tag pointing at the latest attempt. Force-pushing a tag on a pre-release repo is fine, but it would be catastrophic on a tag anyone else has pulled. Worth establishing: tags under `v0.0.x` / `v0.1.x` are mutable pre-release; tags `v1.0.0+` become immutable when the public release happens.
- **MSYS path-mangling on Git Bash for Windows** bit us once (`openssl req -subj "/emailAddress=..."` got mangled into `/c/Program Files/Git/emailAddress=...`). Fix: prefix with `MSYS_NO_PATHCONV=1`. Worth adding to our CLAUDE.md as a Windows gotcha — it'll bite again.
- **OpenSSL 3.5.5 on Windows handled CSR + cert bundling cleanly**, including the `-legacy` flag needed for CodeMagic-compatible `.p12` files. No Mac was needed for any of the cryptographic work. Confirms: a MacBook is only required for interactive Xcode debugging, not for the shipping flow.
- **CodeMagic API returned 404 on several endpoints I tried** (`/teams`, `/code-signing-identities/ios`). Their REST API is narrower than the UI — can trigger + monitor builds, fetch logs, list apps, but not introspect team-level signing state. Worth noting for future debugging sessions.

## What to do differently next session

1. **Time-box first-run CI-tool setup.** If a documented "automatic" feature hasn't worked after 20 minutes of configuration attempts, fall back to the fully-manual path. Don't iterate five times.
2. **Pre-read the user's frustration.** A single "why is this so hard?" from the user is the earlier signal. If I'm on iteration 3 of speculative fixes, proactively pause and ask "do you want me to try a completely different approach?" rather than push a 4th attempt.
3. **Separate noisy experimental commits from clean ones.** When iterating on a CI config, either squash at the end or keep experiments on a throwaway branch. Five force-pushed yaml iterations in the main-branch log is more history than the work deserves.
4. **Add Windows-shell gotchas to CLAUDE.md** as they come up. MSYS path-mangling is the new one from this session.
5. **Write the "working flow" into repo docs immediately after it works**, not at session-end. SIGNING.md's broken auto-flow was the reference both the user and I kept coming back to even after we proved it didn't work. Updating docs mid-session would have saved that confusion.
6. **Use ScheduleWakeup / background-run for long CodeMagic builds**. The in-foreground polling loop worked but consumed context tokens on every sleep iteration. A background monitor with a single completion notification is cheaper.
